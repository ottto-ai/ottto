# Connector Registry

This directory is the manifest-backed source and collector registry root.

The public product and CLI language remains "app". These manifests and
generated artifacts use the internal "source" and "collector" language so
backend, local platform, and policy code can reason about capabilities without
changing existing onboarding contracts.

The v1 contract intentionally does not include `supported_platforms`. Runtime
machine state and operation descriptors own availability.

The connector registry is not the pricing or attribution selector contract.
Collectors may emit records that include `selector_context.v1`, billing
attribution, and cost-source evidence, but manifests must not rename or replace
existing selector fields such as `service_tier`, `speed_mode`, `mode`,
`billing_channel`, `auth_mode`, `gateway_provider`, or
`subscription_product`, nor cost-source values such as `connector_actual`,
`catalog_estimated`, `app_reported`, and `mixed`.

## Package Layout

- `sources/<source>/source.toml` describes one internal source package.
- `sources/<source>/README.md` explains the source package for contributors.
- `sources/<source>/POLICY.md` is the source governance file. It records
  documented surfaces, unsupported or undocumented surfaces, local-only
  behavior, and upload boundaries.
- `sources/<source>/fixtures/` holds source-level parser fixtures, such as
  local session JSONL examples.
- `sources/<source>/collectors/<collector>/collector.toml` describes one
  collector capability.
- `sources/<source>/collectors/<collector>/fixtures/*.json` proves the
  collector's emitted record names and redaction/upload policy.
- `registry.generated.json` is generated by
  `cd backend && uv run python scripts/generate_connector_registry.py`.
- `schemas/*.schema.json` documents the closed manifest, registry, and fixture
  contracts.

## Source Manifest Fields

`source.toml` uses `schema_version = "source_manifest.v1"` and is a closed
contract. Unknown fields fail generation and backend loading. Every field
listed below is required; use an explicit empty array when a source has no
collector ids yet.

- `source_id`: canonical internal source id.
- `app_slug`: product app slug this source backs.
- `display_name`: human-readable product name.
- `publisher`: owner or publishing namespace.
- `review_tier`: `official`, `ottto_labs`, `reviewed_community`,
  `community`, or `custom_local`.
- `maturity`: `stable`, `beta`, `experimental`, `undocumented_surface`,
  `writes_config`, or `local_only`.
- `operations`: supported operation verbs, such as `detect`, `verify`,
  `repair`, `collect_usage`, `monitor_quota`, `upload_snapshot`, and
  `diagnostics`.
- `collectors`: collector ids that must have matching collector manifests.

## Collector Manifest Fields

`collector.toml` uses `schema_version = "collector_manifest.v1"` and is also a
closed contract. Every field listed below is required, including `emits`;
active collectors should list every record family they produce.

- `source_id` and `collector_id`: stable ids. The collector `source_id` must
  match its source package.
- `display_name`: label shown in internal diagnostics and UI collector details.
- `operations`: collector-level operation verbs.
- `data_source_kind`: `local_enriched`, `live_telemetry`,
  `integration_connector`, or `cloud_billing_connector`.
- `default_state`: `enabled`, `requires_setup`, or `disabled` before org policy
  overrides are applied.
- `review_tier` and `maturity`: collector-specific governance posture.
- `risk_classes`: explicit safety labels for default-policy review. Valid
  values are `auth_adjacent`, `network_calls`, `hidden_credential_read`, and
  `raw_prompt_or_output`; use `[]` when none apply.
- `uploads_raw_content`: whether the collector is allowed to upload raw
  prompts, responses, command output, tool output, local paths, or secrets. The
  checked-in official collectors set this to `false`.
- `emits`: record families the collector can produce, for example
  `local_usage_snapshots`, `telemetry_records`, `agent_status_snapshots`, and
  `source_plan_profiles`.

## Fixture Expectations

Every collector needs at least one fixture JSON file under
`sources/<source>/collectors/<collector>/fixtures/`.

The fixture contract is `collector_fixture.v1` and must include:

- matching `source_id` and `collector_id`;
- optional `input_fixture_paths` relative to the fixture file, pointing at
  local parser inputs when the collector reads source-level fixture material;
- `emitted_records`, with one sample object for every manifest `emits` value;
- `upload_policy.uploads_raw_content`, matching the collector manifest;
- `upload_policy.boundary`, written as a concise human-readable upload boundary;
- `upload_policy.redacts`, including prompt, response, tool output, command
  output, local path, and credential classes when raw content is disallowed.

The fixture test rejects samples that expose raw prompt/response fields, command
or tool output, local paths, credentials, cookies, API keys, passwords, secrets,
or raw content fields.

## SDK And Testkit Helpers

The top-level Rust crates under `crates/` expose reusable helper APIs for
contributors:

- `ottto-connector-sdk` owns schema-version constants, vocabulary parsers,
  manifest id and emitted-record validators, and the shared redaction
  vocabulary for safe collector fixtures.
- `ottto-connector-testkit` owns contract assertion helpers for source
  manifests, collector manifests, fixture upload policy, emitted-record names,
  duplicate ids, forbidden v1 fields, and raw-content sample keys.
- `ottto-connector-testkit/tests/first_party_sources.rs` exercises the
  checked-in Codex, Claude Code, and Pi packages with those public helper APIs,
  so the first-party packages stay usable as contributor examples.

Use the testkit in source package tests instead of copying backend generator
logic:

```rust
use ottto_connector_testkit::{
    assert_collector_manifest_contract, CollectorManifestContract,
};

let manifest = CollectorManifestContract {
    source_id: "codex",
    collector_id: "local_sessions",
    display_name: "Local sessions",
    operations: &["collect_usage", "upload_snapshot", "diagnostics"],
    data_source_kind: "local_enriched",
    default_state: "enabled",
    review_tier: "official",
    maturity: "stable",
    risk_classes: &[],
    uploads_raw_content: false,
    emits: &["local_usage_snapshots"],
    fields: &[
        "schema_version",
        "source_id",
        "collector_id",
        "display_name",
        "operations",
        "data_source_kind",
        "default_state",
        "review_tier",
        "maturity",
        "risk_classes",
        "uploads_raw_content",
        "emits",
    ],
};

assert_collector_manifest_contract(&manifest).unwrap();
```

## Review Tiers

Review tiers describe who maintains a source or collector and how conservative
default enablement should be:

- `official`: maintained by Ottto, allowed to default on when the manifest says
  `enabled`.
- `ottto_labs`: maintained by Ottto but experimental; default on only for
  pilot/design-partner organizations.
- `reviewed_community`: external or community contribution that has passed
  review; opt-in unless org policy enables the tier.
- `community`: community maintained and opt-in only.
- `custom_local`: organization-local extension, never enabled globally by
  manifest alone.

## Default Policy Matrix

The org-level default policy matrix for collectors is intentionally outside the
manifest files. Public source packages declare conservative defaults; each
consumer applies its own organization policy on top:

- `official` collectors are on by default, subject to manifest `default_state`.
- `ottto_labs` collectors are on by default only for
  `pilot_design_partner` organizations.
- `reviewed_community`, `community`, and `custom_local` collectors are opt-in
  unless org policy enables the review tier or a specific `source:collector`
  ref.

Advisor may reason over evidence only from collectors that are enabled by this
matrix. Sensitive Advisor execution is controlled separately by org-level
standing approvals; connector manifests do not grant action permissions.
Existing source telemetry opt-outs still gate ingestion at the telemetry
settings boundary.

The registry generator also enforces collector safety before the org-level
matrix runs. `default_state = "enabled"` is rejected for collectors with the
`hidden_credential_read`, `raw_prompt_or_output`, or `network_calls` risk
classes, `live_telemetry` data-source kind, `writes_config` maturity, or the
combination of `undocumented_surface` maturity with `auth_adjacent` or
`network_calls`. Use `requires_setup` or `disabled` for those collectors so
hidden credential reads, private endpoints, network probes, config writes, and
raw-content upload cannot turn on by default.

## Generated Registry Flow

`connectors/registry.generated.json` is the checked-in public registry artifact
for source packages and collectors. Changing manifests without updating the
generated registry is incomplete. Public contributors should validate the
manifest and fixture contracts with the Rust testkit before submitting source
package changes:

```bash
PATH="$HOME/.cargo/bin:$PATH" \
  cargo test --manifest-path crates/Cargo.toml \
    -p ottto-connector-testkit \
    --test first_party_sources
```

`connectors/registry.generated.json` is consumed by:

- cloud services that expose source descriptors;
- local platform status and CLI JSON surfaces that expose collector details;
- product app/source UI that preserves existing onboarding language while
  showing collector-level metadata where intended;
- private consumer repositories that derive runtime registries from this public
  source package contract.

Private consumers are responsible for their own generated-code checks, service
loading tests, and UI contract tests. Those checks must treat this public
registry as the source of truth rather than redefining source package fields.

## Required Checks

For connector package changes, run the smallest relevant subset first:

```bash
PATH="$HOME/.cargo/bin:$PATH" cargo test --manifest-path crates/Cargo.toml -p ottto-connector-testkit --test first_party_sources
jq empty connectors/registry.generated.json schemas/*.schema.json
```
